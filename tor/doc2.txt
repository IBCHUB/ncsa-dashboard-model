เอกสารสรุปข้อมูล: โครงการบูรณาการและเชื่อมโยงระบบ Threat Intelligence ของ สกมช.
บทสรุปสำหรับผู้บริหาร (Executive Summary)
เอกสารฉบับนี้สรุปสาระสำคัญจากการประชุมเก็บความต้องการ (Requirement) สำหรับโครงการบูรณาการและเชื่อมโยงระบบ Threat Intelligence ของสำนักงานคณะกรรมการการรักษาความมั่นคงปลอดภัยไซเบอร์แห่งชาติ (สกมช.) โดยมีวัตถุประสงค์หลักเพื่อพัฒนาระบบศูนย์กลางข้อมูลภัยคุกคามทางไซเบอร์ของประเทศไทยภายใต้ชื่อ "Thailand Cyber Intelligent" (TCI) โครงการนี้แบ่งออกเป็น 5 กลุ่มงานหลัก ได้แก่  การดำเนินงาน (Operation) ,  การพัฒนาระบบ (System Development) ,  การจัดหา (Procurement) ,  การฝึกอบรม (Training) , และ  บุคลากร (Personnel)สถาปัตยกรรมทางเทคนิคของโครงการจะใช้  ELK Stack (Elasticsearch)  เป็นแกนหลักในการประมวลผลข้อมูลผ่านกระบวนการ 3 ขั้นตอนสำคัญ คือ  1) Raw Data Lake  สำหรับรวบรวมข้อมูลดิบ,  2) Processed Data Lake  สำหรับข้อมูลที่ผ่านการตรวจสอบและคัดกรอง, และ  3) Data Warehouse  สำหรับข้อมูลที่ผ่านการวิเคราะห์เชิงลึกด้วยปัญญาประดิษฐ์ (AI/ML) เพื่อเตรียมพร้อมสำหรับการแสดงผลประเด็นสำคัญที่ได้จากการประชุมคือการกำหนดข้อกำหนดที่ชัดเจนสำหรับการยกระดับความสามารถของระบบ โดยเฉพาะอย่างยิ่ง  กระบวนการเพิ่มความน่าเชื่อถือของข้อมูล (Enrichment)  ที่ต้องสามารถตรวจสอบข้อมูลภัยคุกคาม (IOC) กับแหล่งข้อมูลภายในทั้งหมดของ สกมช. เพื่อสร้างคะแนนความเสี่ยงที่แม่นยำ นอกจากนี้ยังมีความต้องการในการปรับปรุง  แดชบอร์ดแสดงผล (Dashboard)  ให้มีข้อมูลเชิงสถิติที่ลึกซึ้งยิ่งขึ้น เช่น สถิติภัยคุกคาม 10 อันดับแรก, การกรองข้อมูลตามช่วงเวลา และการปรับปรุงตรรกะของ "ตัวชี้วัดระดับภัยคุกคาม" ของประเทศให้มีความน่าเชื่อถือและสามารถอธิบายได้ ในด้านการดำเนินงาน ได้มีการปรับเปลี่ยนแนวทางการติดตั้งระบบ MSP ให้กับหน่วยงานต่างๆ โดยจะเน้น  "การฝึกอบรมหน้างาน (Training on the job)"  เพื่อให้มั่นใจว่าหน่วยงานสามารถใช้งานระบบได้อย่างเต็มประสิทธิภาพท้ายที่สุด ที่ประชุมได้ระบุความเสี่ยงที่ต้องติดตามอย่างใกล้ชิด ได้แก่  ความซับซ้อนในการย้ายระบบ Central Management เดิม  ซึ่งเชื่อมต่อกับเซ็นเซอร์จำนวนมาก และ  ความล่าช้าในการเชื่อมต่อข้อมูลกับหน่วยงานต่างประเทศ
1. ภาพรวมขอบเขตโครงการ
โครงการถูกแบ่งออกเป็น 5 กลุ่มงานหลักเพื่อให้ครอบคลุมทุกมิติ ตั้งแต่การวางโครงสร้างพื้นฐานไปจนถึงการพัฒนาบุคลากร
1.1 การดำเนินงาน (Operation)
* ติดตั้งและเชื่อมต่อแพลตฟอร์ม MSP:  ดำเนินการติดตั้งระบบให้กับ 20 หน่วยงาน พร้อมแนวทางการดำเนินงานใหม่ที่เน้นการจัดประชุม Kick-off และการฝึกอบรมหน้างาน (Training on the job) เพื่อแก้ปัญหาการไม่ใช้งานระบบหลังการติดตั้ง
* จัดทำ Asset Inventory:  สร้างฐานข้อมูลสินทรัพย์ดิจิทัลที่อยู่ในระบบ รวมถึงตำแหน่งที่ตั้ง (เช่น Cloud) และผู้รับผิดชอบ
* เชื่อมต่อแพลตฟอร์มต่างประเทศ:  ดำเนินการเชื่อมโยงข้อมูลกับหน่วยงานในต่างประเทศ
* พัฒนาส่วนวิเคราะห์ข้อมูลภัยคุกคาม:  ติดตั้งเซ็นเซอร์และแม่ข่าย (Server) เพิ่มเติมในระบบ
* ดูแลรักษาระบบ:  ให้บริการดูแลและบำรุงรักษาการเชื่อมโยงของระบบโดยรวม
1.2 การพัฒนาระบบ (System Development)
เป็นการพัฒนา 5 ระบบย่อยซึ่งทำงานเชื่อมโยงกันบนสถาปัตยกรรม ELK
1. ระบบ Big Data และ Data Analytic:  พัฒนาส่วนวิเคราะห์ข้อมูลขนาดใหญ่โดยใช้ AI และ ML เพื่อวิเคราะห์และสร้าง Threat & Security Intelligence
2. ระบบศูนย์รวมจัดเก็บข้อมูล:  สร้างระบบ Collector เพื่อรวบรวมข้อมูลจากแหล่งต่างๆ เช่น Threat Feed, News, และ Sandbox
3. ระบบเชื่อมโยงและตรวจสอบข้อมูล (Validation):  นำข้อมูลที่รวบรวมได้มาตรวจสอบความถูกต้อง (Validate) และเพิ่มความน่าเชื่อถือ (Enrich)
4. ระบบทดสอบโปรแกรมต้องสงสัย (Sandbox):  พัฒนาและติดตั้งระบบ Sandbox เพื่อวิเคราะห์มัลแวร์และโปรแกรมต้องสงสัย
5. ระบบคลังข้อมูล (Data Warehouse):  จัดทำคลังข้อมูลที่พร้อมใช้งาน เพื่อนำข้อมูลไปแสดงผลในรูปแบบ Dashboard หรือส่งต่อให้ระบบอื่นๆ
1.3 การจัดหา (Procurement)
* แพลตฟอร์ม MSP:  จัดหาแพลตฟอร์มสำหรับให้บริการหน่วยงานต่างๆ
* Threat Intelligence Feed:  จัดหาข้อมูลจากผู้ให้บริการภายนอก (ระบุว่าใช้ Recorded Future)
* พื้นที่และบริการอินเทอร์เน็ต:  จัดหาพื้นที่วางตู้แร็คและวงจรอินเทอร์เน็ต
* Threat Intelligence Tool:  จัดหาเครื่องมือเพิ่มเติม (ระบุว่าใช้ Tanium)
* VMware License:  จัดหาลิขสิทธิ์สำหรับใช้งาน
* เครื่องคอมพิวเตอร์แม่ข่าย:  จัดหาฮาร์ดแวร์จำนวน 3 ชุด
1.4 การฝึกอบรม (Training)
1. จัดอบรม 3 หลักสูตร:
2. Workshop MSP
3. Forensic Search
4. หลักสูตรสำหรับเจ้าหน้าที่ผู้ดูแลระบบของ สกมช. จำนวน 5 คน
1.5 บุคลากร (Personnel)
* จัดหาบุคลากรสำหรับโครงการ ประกอบด้วย:
* หัวหน้าโครงการ
* นักวิเคราะห์ข้อมูล
* ผู้บริหารโครงการ
2. สถาปัตยกรรมและโครงสร้างทางเทคนิค
2.1 โครงสร้างพื้นฐาน (Infrastructure)
* ฮาร์ดแวร์:  จะมีการติดตั้งเครื่องแม่ข่ายใหม่ 3 ชุด ที่ GDC โดยจะติดตั้งในตู้เดิมของระบบ MC เพื่อให้ง่ายต่อการบูรณาการ
* ระยะเวลา:  คาดว่าฮาร์ดแวร์จะถูกส่งมอบภายใน 45 วัน (ประมาณต้นเดือนกุมภาพันธ์)
* แผนสำรอง:  หากฮาร์ดแวร์ล่าช้า ทาง สกมช. สามารถจัดสรรพื้นที่บนฮาร์ดแวร์เดิมให้ใช้งานชั่วคราวได้ เพื่อให้การพัฒนาไม่หยุดชะงัก
2.2 สถาปัตยกรรมข้อมูล (Data Architecture)
ระบบถูกออกแบบให้มีการประมวลผลข้อมูลเป็น 3 ขั้นตอนหลักบนพื้นฐานของ ELK Stack
1. Raw Data Lake:  เป็นถังเก็บข้อมูลดิบแรกสุดที่รับข้อมูลจาก Collector ทุกแหล่ง
2. Processed Data Lake:  ข้อมูลจากถังแรกจะถูกนำมาประมวลผล, คัดกรองข้อมูลขยะ (Cleansing), และตรวจสอบความถูกต้อง (Validation) ก่อนจะถูกจัดเก็บในถังนี้ ซึ่ง  ถังข้อมูลนี้คือเป้าหมายหลักในการสำรองข้อมูล (Backup)
3. Data Warehouse:  ข้อมูลที่ผ่านการคัดกรองแล้วจะถูกส่งมาที่นี่เพื่อประมวลผลด้วย AI/ML (เช่น การจัดกลุ่ม - Classification) ก่อนจะถูกนำไปจัดเก็บเพื่อรอการแสดงผลบน Dashboard
2.3 แหล่งข้อมูลและการรวบรวม (Data Sources and Collection)
ระบบจะรวบรวมข้อมูลจากหลากหลายแหล่งที่มาผ่าน Collector ที่แตกต่างกัน | แหล่งข้อมูล | วิธีการรวบรวม | หมายเหตุ | | :--- | :--- | :--- | |  Threat Feeds / Web Forums  | สร้าง Node เฉพาะสำหรับรัน  Python Script  เพื่อดึงข้อมูล (Web Scraping) | เป็นการดึงข้อมูลจากแหล่งข้อมูลฟรีเป็นหลัก | |  XDR / Sandbox  | ใช้  API ของ Elastic  ในการเชื่อมต่อและดึงข้อมูลโดยตรง | | |  Honeypot  |  อ่านไฟล์ Log  ที่เป็น Text File จากระบบ Elastic ของ Honeypot โดยตรง | เป็นการเชื่อมต่อกับระบบ Honeypot จริง ซึ่งยังไม่เคยทดสอบและเป็นจุดที่ต้องให้ความสำคัญ | |  แพลตฟอร์มกลาง (สกมช.)  | เชื่อมต่อผ่าน  API  เพื่อดึงข้อมูลที่รวบรวมไว้แล้วจากแพลตฟอร์มกลาง | เป็นแหล่งข้อมูลเพิ่มเติมที่จะถูกนำเข้ามาในโครงการนี้ |
3. ประเด็นสำคัญและข้อกำหนดหลักจากที่ประชุม
3.1 กระบวนการตรวจสอบและเพิ่มความน่าเชื่อถือของข้อมูล (Data Validation & Enrichment)
นี่เป็นหัวใจสำคัญของระบบ โดยมีการกำหนดแนวทางที่ชัดเจนดังนี้
* การสกัด IOC:  ระบบจะต้องสามารถสกัดข้อมูลบ่งชี้ภัยคุกคาม (IOCs) เช่น IP, URL, Hash ออกจากข้อมูลดิบได้
* การตรวจสอบข้ามแหล่งข้อมูล (Cross-Checking):   ข้อกำหนดที่สำคัญที่สุด  คือ แทนที่จะตรวจสอบ IOC กับ Threat Intelligence ภายนอกเพียงอย่างเดียว ระบบจะต้องนำ IOC ไปตรวจสอบกับ  "แหล่งข้อมูลภายในทั้งหมด"  ที่มีอยู่ (เช่น ข้อมูลจาก XDR, Honeypot, แพลตฟอร์มกลาง)
* การให้คะแนนความเสี่ยง (Scoring):  IOC ที่ปรากฏในหลายแหล่งข้อมูลภายในจะถูกให้คะแนนความเสี่ยงสูงกว่า IOC ที่เจอในแหล่งข้อมูลเดียว ซึ่งจะทำให้ผลลัพธ์น่าเชื่อถือมากขึ้น
* การอ้างอิงแหล่งที่มา:  ทุก IOC ที่ผ่านการประมวลผลแล้วต้องสามารถระบุแหล่งที่มาที่ตรวจสอบพบได้อย่างชัดเจน
3.2 ระบบปัญญาประดิษฐ์ (AI System)
* วัตถุประสงค์:  หน้าที่หลักของ AI ในโครงการนี้คือ  การจัดกลุ่มประเภทของภัยคุกคาม (Threat Classification)
* อัลกอริทึม:  ในเบื้องต้นมีการเสนอใช้อัลกอริทึม เช่น Logistic Regression และ Random Forest
* ข้อเสนอแนะ:  ที่ประชุมเสนอให้มีการทำงานร่วมกันระหว่างทีมพัฒนาและทีมนักวิเคราะห์ข้อมูล/AI ของ สกมช. เพื่อเลือกและปรับปรุงโมเดลให้เหมาะสมกับข้อมูลจริง
3.3 แดชบอร์ดและระบบแสดงผล "Thailand Cyber Intelligent"
แดชบอร์ดเป็นส่วนที่ผู้ใช้งานจะโต้ตอบโดยตรง และมีข้อกำหนดในการปรับปรุงหลายประการ
* ฟังก์ชันการค้นหา:
* ต้องรองรับการค้นหา IOC ทุกประเภท (MD5, SHA1, SHA256)
* เมื่อผู้ใช้ค้นหา ระบบจะต้องนำค่าที่ได้ไปตรวจสอบกับ  "ทุกแหล่งข้อมูล"  ที่มี ไม่ใช่การเลือกตรวจสอบเฉพาะแหล่ง
* ผลการค้นหาต้องแสดง  คะแนนความเสี่ยง  และ  แหล่งข้อมูลอ้างอิง  อย่างชัดเจน
* การแสดงผลเชิงสถิติ:
* ข้อมูลภาพรวม:  ต้องมีตัวเลขสรุปจำนวน IOC ทั้งหมด และจำแนกตามประเภท
* สถิติ Top 10:  แสดงรายการ 10 อันดับแรกของภัยคุกคามที่พบบ่อย หรือมีคะแนนความเสี่ยงสูงสุด
* Drill-Down:  ผู้ใช้ต้องสามารถคลิกจากข้อมูลสรุป (เช่น Top 10) เพื่อเข้าไปดูรายละเอียดเพิ่มเติมได้ (อาจมีระบบควบคุมสิทธิ์การเข้าถึงข้อมูลเชิงลึก)
* การกรองข้อมูล (Filter):  ต้องมีฟังก์ชันกรองข้อมูลตามช่วงเวลาที่ยืดหยุ่น (รายวัน, รายเดือน, รายไตรมาส, หรือกำหนดเอง)
* ตัวชี้วัดระดับภัยคุกคามของประเทศ:
* ตรรกะเดิมที่ใช้การเปรียบเทียบข้อมูลแบบวันต่อวันนั้น  "ยังไม่เหมาะสม"
* ข้อเสนอแนะ:  ให้เปลี่ยนไปใช้เกณฑ์ที่อิงตาม  "ปริมาณภัยคุกคามที่กระทบต่อแต่ละ Sector"  เช่น หาก Sector การเงินถูกโจมตีหนัก ระดับภัยคุกคามของ Sector นั้นก็จะแสดงเป็นสีแดง ซึ่งเป็นตรรกะที่สมเหตุสมผลและอธิบายได้
* ฟังก์ชันการสร้างรายงาน:  ควรมีปุ่มสำหรับสร้างรายงานอย่างง่าย เช่น Export รายชื่อ IOCs ประจำวัน/สัปดาห์
4. ความเสี่ยงและประเด็นที่ต้องติดตาม
* การย้ายระบบ Central Management (ความเสี่ยงสูง):  ระบบจัดการกลางเดิมมีการเชื่อมต่อกับเซ็นเซอร์เกือบ 40 ตัวที่ติดตั้งในหน่วยงานต่างๆ การย้ายระบบมายังฮาร์ดแวร์ใหม่จำเป็นต้องมีการแก้ไขการตั้งค่า (re-config) ที่เซ็นเซอร์ปลายทางทั้งหมด ซึ่งอาจเกิดความซับซ้อนและต้องมีการประสานงานอย่างใกล้ชิด
* การเชื่อมต่อกับหน่วยงานต่างประเทศ:  ปัจจุบันมีความล่าช้าในการเชื่อมต่อซึ่งเกิดจากฝั่งหน่วยงานต่างประเทศยังไม่พร้อม ถือเป็นปัจจัยภายนอกที่อาจกระทบต่อไทม์ไลน์ของส่วนงานนี้
* การตรวจสอบสิทธิ์การใช้งาน Threat Intelligence:  ต้องตรวจสอบเงื่อนไขของ License (เช่น Recorded Future) ว่าสามารถใช้งานในลักษณะ Account เดียวเพื่อทำ Enrichment ในระบบนี้ได้หรือไม่ โดยไม่ขัดกับสิทธิ์การใช้งานของระบบ MSP เดิม
* การทดสอบเชื่อมต่อ Honeypot:  การเชื่อมต่อกับระบบ Honeypot จริงยังไม่เคยมีการทดสอบ จำเป็นต้องมีการทดสอบเพื่อให้แน่ใจว่าสามารถดึงข้อมูลได้ถูกต้องและมีเสถียรภาพ
